## Context

The repository already contains SvelteKit, Better Auth configured with email/password authentication, Drizzle ORM, PostgreSQL, a Better Auth request handler, and session types in `App.Locals`. The generated Better Auth schema is still a placeholder, and the generated demonstration login and protected routes are no longer present in the working tree.

This change completes only the smallest coherent authentication slice. Better Auth's user is an authentication identity; it must not become the long-term HRIS Person model. Conversely, the initial Person model must not speculate about employment, demographic, organization, or profile fields that have not yet been specified.

## Goals / Non-Goals

**Goals:**

- Provide working email/password login, persisted session resolution, one protected page, and logout.
- Provide a ready-to-use default login in development without provisioning an account or exposing its credentials in production.
- Introduce a product-owned Person with a stable UUID and an explicit one-to-one authentication link.
- Ensure every authenticated account used by the application resolves to one Person.
- Use the existing dependencies and SvelteKit form/action patterns.
- Cover the flow with focused server and browser tests.

**Non-Goals:**

- Add a registration page or other end-user account provisioning interface.
- Define or implement production account provisioning.
- Add password reset, email verification, social login, MFA, SSO, or account management.
- Add roles, permissions, organizations, employments, teams, reporting, profiles, or other HRIS concepts.
- Add a UI framework, validation framework, test container, script runner, or any other dependency.
- Generalize the protected page into a dashboard or product feature.

## Decisions

### Retain Better Auth and the existing SvelteKit integration

Better Auth continues to own credentials, password hashing, authentication users, accounts, sessions, cookies, and auth endpoints. The existing Drizzle adapter and `sveltekitCookies` integration remain the mechanism for persistence and cookie propagation.

Alternative: implement credentials and sessions directly in SvelteKit. Rejected because the repository already has a purpose-built authentication implementation and custom credential handling would add security risk without product value.

### Add a deliberately minimal Person table

The product-owned `person` table contains:

- `id`: UUID primary key generated by PostgreSQL.
- `auth_user_id`: nullable, unique foreign key to the Better Auth user identifier.
- `created_at` and `updated_at`: timezone-aware audit timestamps.

The link is unique in both directions: one authentication user maps to at most one Person, and one Person maps to at most one authentication user. The foreign key uses `ON DELETE SET NULL` so a future HRIS Person is not destroyed merely because login access is removed. No name or email is duplicated because Better Auth owns those values for this slice, and no employment fields are introduced.

Alternative: add `person_id` to Better Auth's user table. Rejected because generated authentication tables should remain provider-owned and future people without application accounts must remain representable.

### Provision or repair the Person link at the authentication boundary

A small server-only Person repository exposes an idempotent `ensurePersonForAuthUser` operation. Better Auth account provisioning invokes it after user creation when accounts are created through the authentication API. Session resolution also calls the same idempotent operation for an existing valid account, which safely backfills accounts created before this migration.

The database unique constraint resolves concurrent first-session attempts: the operation inserts when absent and then reads the single linked Person. The hook and request code do not add employment or authorization behavior.

Alternative: require a manual Person row before login. Rejected because it permits a valid authentication session that the product cannot associate with its own identity.

### Use SvelteKit server actions for login and logout

`/login` has a server load that redirects an existing session to `/protected` and a default form action that validates the presence of email and password before calling `auth.api.signInEmail`. Authentication failure returns one generic message and never returns the password.

`/protected` has a server load that requires both a valid session and linked Person, then returns only the minimal display data needed to prove the flow. Its logout form action calls `auth.api.signOut` and redirects to `/login`.

The protected page is intentionally not a dashboard or Person profile. Route protection is enforced on the server load, not only through client navigation.

### Keep account creation outside this UI change

No `/register` route or sign-up form is added. Tests may provision an account through Better Auth's server API. This proposal does not establish public registration policy.

### Provision one predictable account in development only

When SvelteKit reports that the application is running in development mode, the server provisions `dev@org-teams.local` with password `password` through Better Auth on the first request. Provisioning is memoized for the running server process and treats an already-existing account as success, making repeated requests and server restarts idempotent. The existing Better Auth user-creation hook provisions the linked Person through the same path as any other authentication account.

Better Auth account provisioning does not automatically create a session. This prevents the first anonymous visitor from receiving the seeded account's session cookie. The `/login` server load returns the default credentials for display only when the same development-mode flag is true. Outside development mode, the provisioning operation is a no-op and the login page returns no default credentials.

Alternative: require every developer or browser test to run a separate account-fixture command. Rejected because a dev-only account at the authentication boundary keeps local setup immediate and repeatable without adding another script or dependency.

### Commit one migration containing auth and Person persistence

Generate the Better Auth schema into the existing auth schema module, export the Person table from the Drizzle root schema, and create a reviewed Drizzle migration. The migration is applied through the existing `db:migrate` command; schema push is not part of this change.

## Risks / Trade-offs

- [Person provisioning after authentication-user creation is not one cross-library transaction] -> Make provisioning idempotent, backfill on valid session resolution, and enforce the one-to-one link in PostgreSQL.
- [A nullable auth link allows a Person without login] -> This is intentional for the future HRIS model; protected access still requires a linked authentication user.
- [Generic login errors provide less diagnostic detail] -> Prefer avoiding account enumeration and log only safe server-side context.
- [The existing Better Auth configuration exposes capabilities beyond the new UI] -> Do not add routes or product affordances for them; configuration hardening can be proposed when registration policy is defined.
- [The default development credentials are predictable and unsafe for production] -> Gate both provisioning and login-page exposure on SvelteKit's development-mode flag, and verify a production build returns neither behavior.
- [Browser tests require a real PostgreSQL database] -> Use the existing Compose PostgreSQL service and committed migrations; add no container-testing dependency.

## Migration Plan

1. Generate and review Better Auth's Drizzle schema.
2. Add the Person schema and idempotent authentication link repository.
3. Generate and apply the combined migration to a clean local PostgreSQL database.
4. Add session/Person resolution to the existing server hook.
5. Add the dev-only default account provisioning and credential guidance.
6. Add `/login` and `/protected`, followed by focused tests.

Rollback removes the two routes, dev-account provisioning, and hook behavior, then reverts the new migration in development by recreating the pre-product database. There is no production product data to migrate.

## Open Questions

- How production accounts will be provisioned is intentionally deferred; this change defines login but not registration policy.
